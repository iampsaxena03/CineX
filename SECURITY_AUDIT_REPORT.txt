SECTION 1: CRITICAL VULNERABILITIES
================================================================================

1. HARDCODED CREDENTIALS IN .ENV FILE
--------------------------------------------------------------------------------
Location: .env (lines 8-16)
Severity: CRITICAL
Description:
  - ADMIN_USERNAME=whoareyou (plaintext)
  - ADMIN_PASSWORD=mynameisjoker@8449 (plaintext)
  - JWT_SECRET=super_secret_key_for_cinexp_admin_1234567890 (weak/guessable)
  - GPLINKS_API_TOKEN=e8960ac8e77431af6f8643ff3728aea32d4e1fc8 (plaintext)
  - EXEIO_API_TOKEN=3a82cc56b2762a23b440b033bc854b5d7fbd38bb (plaintext)
  - DATABASE_URL with credentials exposed

Impact: Full admin compromise, database exposure, API abuse
Remediation:
  - Rotate all credentials immediately
  - Use secrets manager (Vercel env vars with encryption)
  - Generate cryptographically random 256-bit JWT secret

--------------------------------------------------------------------------------

2. NO RATE LIMITING ON ADMIN API ROUTES
--------------------------------------------------------------------------------
Location: src/proxy.ts
Severity: CRITICAL
Description:
  - Only /api/admin/auth POST is rate-limited (5 attempts/15 min in-memory)
  - All other /api/admin/* routes have no rate limiting
  - In-memory rate limit resets on server restart
  - Not suitable for serverless/edge deployments

Impact: Brute-force attacks, DoS, automated exploits
Remediation:
  - Implement distributed rate limiting (Redis/cloud provider)
  - Apply rate limiting to ALL admin endpoints
  - Add progressive penalties (IP blocks)

--------------------------------------------------------------------------------

3. WEAK SESSION VALIDATION IN PROXY MIDDLEWARE
--------------------------------------------------------------------------------
Location: src/proxy.ts:97-106
Severity: CRITICAL
Description:
  - Uses isValidSession() which only checks JWT signature + expiry
  - Does NOT check database session revocation status
  - Allows access with revoked sessions until JWT expires (3 hours)
  - Compare: src/lib/guard.ts uses isValidSessionFull() (proper)

Impact: Bypass session revocation, continued access after logout
Remediation:
  - Change proxy to use isValidSessionFull() from auth.ts
  - Or implement session blacklist

SECTION 2: HIGH SEVERITY ISSUES
================================================================================

4. NO INPUT VALIDATION ON API PARAMETERS
--------------------------------------------------------------------------------
Location:
  - src/app/api/admin/home/sections\[sectionId]\items\route.ts:62
  - src/app/api/admin/downloads\route.ts (multiple locations)
  - src/app/api/download/routes.ts (multiple locations)
Severity: HIGH
Description:
  - parseInt() without bounds checking
  - No validation for tmdbId, seasonNumber, episodeNumber
  - Trusting user input directly in DB queries

Impact: Unexpected behavior, potential data corruption
Remediation:
  - Add explicit validation:
    - tmdbId: positive integer only, max 999999
    - seasonNumber: 1-99
    - episodeNumber: 1-9999

--------------------------------------------------------------------------------

5. NO HTTPS ENFORCEMENT
--------------------------------------------------------------------------------
Location:
  - src/lib/auth.ts:144 (secure: true assumes HTTPS)
  - No explicit HTTP->HTTPS redirects
Severity: HIGH
Description:
  - Assumes HTTPS is always enabled
  - No forced redirect for HTTP requests in app
  - Rely on Vercel/Cloudflare config only

Impact: Downgrade attacks, cookie theft on HTTP
Remediation:
  - Add explicit HTTP->HTTPS redirect middleware
  - Configure HSTS headers

--------------------------------------------------------------------------------

6. CORS EXPOSED DEV ORIGINS IN PRODUCTION
--------------------------------------------------------------------------------
Location: next.config.ts:27
Severity: HIGH
Description:
  - allowedDevOrigins: ["192.168.29.203", "localhost:3000"]
  - Dev IPs exposed in production build

Impact: Potential SSRF, internal network scanning
Remediation:
  - Move allowedDevOrigins to environment-specific config
  - Use separate next.config for dev/prod

SECTION 3: MEDIUM SEVERITY ISSUES
================================================================================

7. IN-MEMORY RATE LIMITING LIMITATIONS
--------------------------------------------------------------------------------
Location: src/proxy.ts:6-8
Severity: MEDIUM
Description:
  - Uses Map() which doesn't persist across instances
  - Serverless/replica sets don't share state
  - No true distributed rate limiting

Impact: Bypassable rate limits in scaled deployments
Remediation:
  - Use external Redis for rate limiting
  - Use cloud provider rate limiting (Cloudflare/Vercel)

--------------------------------------------------------------------------------

8. NO CSRF PROTECTION
--------------------------------------------------------------------------------
Location: Admin API routes
Severity: MEDIUM
Description:
  - Relies only on sameSite: strict cookie policy
  - No explicit CSRF tokens for mutations

Impact: Potential CSRF on legacy browsers
Remediation:
  - Add double-submit cookie pattern
  - Implement CSRF tokens for critical endpoints

--------------------------------------------------------------------------------

9. MISSING SECURITY HEADERS ON PUBLIC ROUTES
--------------------------------------------------------------------------------
Location: src/proxy.ts:111-117
Severity: MEDIUM
Description:
  - Security headers only applied to /admin/* routes
  - Public routes lack X-Content-Type-Options, X-Frame-Options, etc.

Impact: Clickjacking, MIME sniffing, XSS
Remediation:
  - Apply headers to all routes via middleware
  - Use Next.js Headers() in route handlers
